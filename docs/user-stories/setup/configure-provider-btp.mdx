# Configure the provider

To configure the BTP provider, you need **a service binding for an SAP Cloud Management service instance**. The service binding must be manually created **in a subaccount belonging to the same global account where you want to orchestrate your resources**.

In the diagram below, Subaccount A contains the binding for an SAP Cloud Management service instance.
Once the provider is set up, you can orchestrate other subaccounts in code, such as Subaccount B and Subaccount C.

:::image-light
![Setup Cloud Management Service](/img/overview-orchestrate-accounts.png)
:::

##  🚧 Prerequisites

- Your organization has a global account on SAP BTP.
- You've [installed the BTP provider on your control plane](/docs/crossplane-provider-btp/docs/user-stories/setup/install-provider-btp).
- You're xxxconnected to your control plane.

## Set up the SAP Cloud Management service

To orchestrate your global account, you need one instance of the SAP Cloud Management service (with the **central** plan) in one of your subaccounts. To allow API access to the service instance, you also need a **service binding**.

:::info Learn more

For more information on the SAP Cloud Management service, see [Account Administration Using APIs of the SAP Cloud Management Service](https://help.sap.com/docs/btp/sap-business-technology-platform/account-administration-using-apis-of-sap-cloud-management-service?locale=en-US&version=LATEST).

For more information on service instances and bindings, see [Service Instances](https://help.sap.com/docs/service-manager/sap-service-manager/service-instances?locale=en-US&version=LATEST) and [Service Bindings](https://help.sap.com/docs/service-manager/sap-service-manager/service-bindings?locale=en-US&version=LATEST). 

For more information on the BTP platform, see [Basic Platform Concepts](https://help.sap.com/docs/btp/sap-business-technology-platform/btp-basic-platform-concepts?locale=en-US&version=LATEST).

:::

To learn how to set up the SAP Cloud Management service, choose your preferred tool from the tabs below.

<Tabs>
  <TabItem value="bpt-cli" label="Using the btp CLI">

    :::note Install the btp CLI

    If you haven't installed the BTP Command Line Interface (btp CLI) yet, follow the guide from the SAP Developer Center: [Get Started with the SAP BTP Command Line Interface (btp CLI)](https://developers.sap.com/tutorials/cp-sapcp-getstarted.html).
    
    :::

		To create a subaccount containing an instance and binding of the Cloud Management service, complete the following steps:

    1. Log in to the CLI using SSO:

        ```sh title="Run in Terminal"
        btp login --sso
        ```

    2. Create a new subaccount:

        ```sh title="Run in Terminal" {1}
        btp create accounts/subaccount --display-name "cloud-management-subaccount" --region <region> --subdomain <subaccount-subdomain>
        ```

        Replace the following:

        - `<region>`: The region in which to create the subaccount
        - `<subaccount-subdomain>`: The subdomain of the subaccount. Must be unique within the defined region <br/><br/>

        <details>
					<summary> View available regions </summary>

					Run the following command to see available regions.

					```shell title="Run in Terminal" {1}
						btp list accounts/available-region --global-account <global-account-subdomain>
					```
					
        </details>

    3. Copy the subaccount ID to the clipboard. 

    4. Create an SAP Cloud Management service entitlement, instance, and binding by running the commands below.

        ```sh title="Run in Terminal" {1}
        export BTP_SUBACCOUNT_ID=<subaccount-id>
        # Create a service entitlement
        btp assign accounts/entitlement --for-service cis --plan central --enable true --to-subaccount $BTP_SUBACCOUNT_ID
        sleep 5
        # Create a service instance
        btp create services/instance --offering-name cis --service my-cis --plan-name central --parameters '{"grantType": "clientCredentials"}' --subaccount $BTP_SUBACCOUNT_ID
        # Create a service binding
        btp create services/binding --name my-cis-binding --instance-name my-cis --subaccount $BTP_SUBACCOUNT_ID
        btp get services/binding --name  my-cis-binding --subaccount $BTP_SUBACCOUNT_ID
        ```

        Replace `<subaccount-id>` with the ID of your newly created subaccount.

    4. From the output of the previous commands, copy the binding credentials to your clipboard. You'll need the credentials to create a Kubernetes secret.

## Create a `Secret` for the SAP Cloud Management service

To allow Crossplane to communicate with the SAP Cloud Management service, you need to create a Kubernetes secret that references the service binding credentials.

<Tabs>
  <TabItem value="programmatically" label="Using the CLI">

    :::note Install `jq`

    You need [`jq`](https://jqlang.github.io/jq/) installed to create a `Secret` using only the CLI.

    :::

    To create a `Secret`, run the commands below.

    ```sh title="Run in Terminal":
    btp --format json get services/binding --name my-cis-binding --subaccount $BTP_SUBACCOUNT_ID  | jq .credentials -r > binding.json
    kubectl create secret generic cis-provider-secret -n default --from-file=data=./binding.json -o json
    ```
  </TabItem>
  <TabItem value="manually" label="Manually">
 1. Copy the following YAML content and replace the value of `stringData.data` with the credentials from your SAP Cloud Management service binding.

      - If you followed the steps in [Set up the SAP Cloud Management service](#set-up-the-sap-cloud-management-service), the binding credentials should already be copied to your clipboard.<br/>
      <br/>

      :::apply-to-mcp
      ```yaml title="examples/provider/cis-provider-secret.yaml" {9-41}
      apiVersion: v1
      kind: Secret
      metadata:
          namespace: default
          name: cis-provider-secret
      type: Opaque
      stringData:
          data: |
              {
                "endpoints": {
                  "accounts_service_url": "...",
                  "cloud_automation_url": "...",
                  "entitlements_service_url": "...",
                  "events_service_url": "...",
                  "external_provider_registry_url": "...",
                  "metadata_service_url": "...",
                  "order_processing_url": "...",
                  "provisioning_service_url": "...",
                  "saas_registry_service_url": "..."
                },
                "grant_type": "client_credentials",
                "sap.cloud.service": "com.sap.core.commercial.service.central",
                "uaa": {
                  "apiurl": "...",
                  "clientid": "...",
                  "clientsecret": "...",
                  "credential-type": "binding-secret",
                  "identityzone": "...",
                  "identityzoneid": "...",
                  "sburl": "...",
                  "subaccountid": "...",
                  "tenantid": "...",
                  "tenantmode": "shared",
                  "uaadomain": "...",
                  "url": "...",
                  "verificationkey": "...",
                  "xsappname": "...",
                  "xsmasterappname": "...",
                  "zoneid": "..."
                }
              }
      ```
      :::

2. Save the updated YAML content to `cis-provider-secret.yaml`.

3. Apply the secret to your control plane:

    ```sh title="Run in Terminal"
    kubectl apply -f cis-provider-secret.yaml
    ```

  </TabItem>

## Create and validate a technical user

To allow Crossplane to communicate with the BTP API, we also need a technical user to be created on BTP.
It's important that this technical user has admin rights to the subaccount so that Crossplane can be able to create and manage resources.

A technical user is a non-human user account created specifically for programmatic access to systems, services, or applications. Unlike regular user accounts, technical users are intended to be used by automated processes and tools rather than individuals.
They are typically granted specific permissions tailored to the tasks they need to perform. It's a good practice to use a technical user for configuring the BTP subaccount instead of individual IDs.

### Validate the technical user

1. Once the technical user is created, please add it to the global account and the subaccount with admin privilege as described in the SAP Help Portal: [Assign User Groups to Role Collections](https://help.sap.com/docs/btp/sap-business-technology-platform/assign-user-groups-to-role-collections?locale=en-US&version=LATEST).
2. Log in to the SAP BTP cockpit using the technical user in an incognito browser window of your choice, ensuring that SSO is disabled.
3. Verify that the technical user can authenticate with the BTP through the btp CLI:

    ```sh title="Run in Terminal" {1}
    btp login --url <btp-environment-url> --user <technical-user-email> --password <technical-user-password> --subdomain <global-account-subdomain>
    ```

    Replace the following:
    - `<btp-environment-url>`: The CLI server URL
    - `<technical-user-email>`: The email of the technical user
    - `<technical-user-password>`: The password of the technical user
    - ` <global-account-subdomain>`: The subdomain of the global account. To be found in the BTP cockpit under **Switch Global Account**


## Create a `Secret` for the technical user

You can create a `Secret` using the technical user (P-user) credentials.

1. Save the following YAML content to `sa-provider-secret.yaml`.

    :::apply-to-mcp
    ```yaml title="sa-provider-secret.yaml" {10-12}
    apiVersion: v1
    kind: Secret
    metadata:
        namespace: default
        name: sa-provider-secret
    type: Opaque
    stringData:
        credentials: |
            {
              "email": "<technical-user-email>",
              "username": "<technical-user-username>",
              "password": "<technical-user-password>"
            }
    ```
    :::

    Replace the following:
    
    - `<technical-user-email>`: The email address of the technical user
    - `<technical-user-username>`: The username of the technical user
    - `<technical-user-password>`: The password of the technical user

    :::note Differences in BTP landscapes

    We noticed that for some BTP landscapes (cn40) the `<technical-user-username>` does not work (resulting in 401 errors when trying to reconciliate e.g. ServiceManager). In this case, the `<technical-user-username>` must be set with the email address of the technical user too.

    :::

2. Apply `sa-provider-secret.yaml` to your control plane:

    ```sh title="Run in Terminal" {1}
    kubectl apply -f sa-provider-secret.yaml
    ```

## Create a `ProviderConfig`

In the `ProviderConfig`, you'll reference the Kubernetes secrets created in [Create a `Secret` for the SAP Cloud Management service](#create-a-secret-for-the-sap-cloud-management-service) and [Create a `Secret` for the technical user](#create-a-secret-for-the-technical-user). To create the `ProviderConfig`, follow the steps below.

1. Save the following YAML content to `config.yaml`:
    <br/>

    :::apply-to-mcp
    ```yaml title="config.yaml" {6}
    apiVersion: btp.sap.crossplane.io/v1alpha1
    kind: ProviderConfig
    metadata:
        name: account-provider-config
    spec:
        globalAccount: <global-account-subdomain>
        cliServerUrl: https://cli.btp.cloud.sap
        cisCredentials:
            secretRef:
                name: cis-provider-secret
                namespace: default
                key: data
            source: Secret
        serviceAccountSecret:
            secretRef:
                key: credentials
                name: sa-provider-secret
                namespace: default
            source: Secret
    ```
    :::

    Replace `<global-account-subdomain>` with the subdomain of your BTP global account.

2. Apply the configuration:

    ```sh title="Run in Terminal"
    kubectl apply -f config.yaml
    ```
