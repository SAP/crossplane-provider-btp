# Advanced: Configure Controller Flags

The BTP provider exposes a set of controller flags that let you tune reconciliation behavior — for example, how aggressively the provider retries failed operations or how often it checks for drift.
These flags are passed as arguments to the provider's container process and can be configured via a [`DeploymentRuntimeConfig`](https://docs.crossplane.io/latest/packages/providers/#runtime-configuration).

## Available Flags

### Reconciliation throughput

| Flag | Default | Description |
|------|---------|-------------|
| `--max-reconcile-rate` | `3` | How many reconciles per second are allowed globally across all controllers. Technically: all controllers share one token bucket that refills at this rate with a burst of 10x. |
| `--max-concurrent-reconciles` | `3` | How many reconciles can run in parallel per controller. Technically: number of worker goroutines dequeuing from the controller-runtime work queue per controller. |

### Polling

| Flag | Default | Description |
|------|---------|-------------|
| `--poll` | `1m` | How often a successfully-reconciled resource is re-checked for drift from the desired state. Technically: requeue delay after any successful reconcile (both when up-to-date and after successful updates).|
| `--poll-jitter` | `0` (disabled) | Random spread added to poll interval to avoid all resources polling at the same instant. Technically: random duration between -jitter and +jitter added to poll interval on each requeue. |
| `--sync` / `-s` | `1h` | How often all watched resources are re-listed from the API server to correct missed watch events. Technically: controller-runtime cache SyncPeriod; triggers a full re-list per resource type, with 10% jitter between controllers. |

### Error handling

| Flag | Default | Description |
|------|---------|-------------|
| `--backoff-base` | `1s` | Initial wait time before retrying a failed reconcile. Technically: base duration for per-item exponential backoff (baseDelay * 2^numFailures) in the controller-runtime work queue. |
| `--backoff-max` | `60s` | Maximum wait time between retries of a failing reconcile. Technically: cap for the exponential backoff rate limiter. |
| `--reconcile-timeout` | `3m` | Maximum time a single reconcile may take before being canceled. Technically: deadline for the externalCtx passed to external API calls; a separate parent context with +30s grace period remains active for K8s status updates after expiry. |

### Kubernetes API client

| Flag | Default | Description |
|------|---------|-------------|
| `--kube-client-qps` | `15` | How many requests per second the provider may send to the Kubernetes API. Technically: token refill rate for the client-go REST client rate limiter (applies to all API calls from this provider). |
| `--kube-client-burst` | 2x `kube-client-qps` | Maximum burst of requests allowed to the Kubernetes API above the steady QPS. Defaults to 2x `--kube-client-qps` if not explicitly set. Technically: bucket size for the client-go REST client rate limiter. |

### Leader election

These flags control the leader election mechanism for high-availability deployments. Only one replica reconciles at a time; others stand by to take over.

| Flag | Default | Description |
|------|---------|-------------|
| `--leader-election` / `-l` | `false` | Enable leader election for the controller manager. Can also be set via the `LEADER_ELECTION` environment variable. |
| `--leader-election-lease-duration` | `60s` | How long standby replicas wait before taking over leadership. Technically: Kubernetes Lease duration; other candidates wait this long before force-acquiring. |
| `--leader-election-renew-deadline` | `50s` | How long the active leader tries to renew before giving up. Technically: deadline for the leader to refresh the Lease; on expiry the leader steps down and cancels in-flight reconciles. |
| `--leader-election-retry-period` | `10s` | How often leader election actions (renew or acquire) are attempted. Technically: interval between LeaderElector client tries; applies to both the leader refreshing its lease and standbys attempting to acquire. |

### Other

| Flag | Default | Description |
|------|---------|-------------|
| `--debug` / `-d` | `false` | Run with debug logging. |

## Exponential Backoff Details

The `--backoff-base` and `--backoff-max` flags control how the provider handles transient errors. When a reconciliation fails, the provider waits before retrying using an exponential backoff strategy:

- After the 1st failure: waits approximately `--backoff-base`
- Each subsequent failure doubles the wait time
- The wait time is capped at `--backoff-max`

Tuning these values is useful when:
- The BTP API is rate-limiting your requests (increase `--backoff-base` and `--backoff-max`)
- You want faster recovery from transient errors in a low-traffic environment (decrease `--backoff-base`)

:::tip
Avoid setting `--backoff-max` to a value that coincides with your `--poll` or `--sync` interval, as this can cause a burst of reconcile requests all firing at the same time. For example, if `--poll=1m`, prefer `--backoff-max=50s` or `--backoff-max=90s` rather than `60s`.
:::

## Configure Flags via a `DeploymentRuntimeConfig`

Crossplane manages the provider's `Deployment` directly. To pass custom arguments to the provider container, use a [`DeploymentRuntimeConfig`](https://docs.crossplane.io/latest/packages/providers/#runtime-configuration) to override the container's `args`.

### Example

The following example configures higher throughput with jitter-based spread, which is useful when the provider manages a large number of resources.

1. Create and apply `DeploymentRuntimeConfig`:

    ```yaml title="provider-btp-runtime-config.yaml"
    apiVersion: pkg.crossplane.io/v1beta1
    kind: DeploymentRuntimeConfig
    metadata:
      name: provider-btp
      namespace: crossplane-system
    spec:
      deploymentTemplate:
        spec:
          selector: {}
          template:
            spec:
              containers:
                - name: package-runtime
                  args:
                    - --max-reconcile-rate=10
                    - --max-concurrent-reconciles=5
                    - --poll=2m
                    - --poll-jitter=10s
                    - --kube-client-qps=50
                    - --backoff-base=5s
                    - --backoff-max=90s
                    - --leader-election
    ```

2. This `DeploymentRuntimeConfig` needs to be referenced by the provider's `Provider` resource.

:::warning

Depending on your setup, the `Provider` resource may already exist and be managed by your platform or tooling — do not modify it in that case. This applies, for example, when using [Open Control Plane (OCPL)](https://github.com/openmcp-project/), where the `Provider` resource is typically managed for you.

:::

```yaml title="provider-btp.yaml"
apiVersion: pkg.crossplane.io/v1
kind: Provider
metadata:
  name: provider-btp
spec:
  package: ghcr.io/sap/crossplane-provider-btp/crossplane/provider-btp:<version>
  runtimeConfigRef:
    name: provider-btp
```

Replace `<version>` with the desired provider version, e.g. `v1.3.0`.
